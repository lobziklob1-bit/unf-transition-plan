# QWEN.md — План перехода на 1С УНФ

## Обзор проекта

Интерактивный лендинг для управления задачами, инструкциями и шагами миграции при переходе компании с Glide на **1С:Управление нашей фирмой (УНФ)**. Данные хранятся в **PostgreSQL**.

### Технологии

| Слой | Стек |
|------|------|
| **Backend** | Node.js + Express 5 |
| **Frontend** | Vanilla JS, HTML, CSS (тёмная тема, адаптивный дизайн) |
| **База данных** | PostgreSQL 16 |
| **Зависимости** | `express`, `pg`, `cors`, `dotenv`, `axios` |

### Архитектура

```
unf-transition-plan/
├── src/                    # Фронтенд (статические файлы)
│   ├── index.html          # Главная страница (лендинг)
│   ├── style.css           # Стили (~1350 строк, CSS variables, анимации)
│   └── app.js              # Логика интерфейса (~1200 строк, vanilla JS)
├── server/
│   └── index.js            # Node.js сервер (Express, ~270 строк)
├── package.json
└── README.md
```

## База данных

- **СУБД:** PostgreSQL 16
- **База:** `unf_transition_plan`
- **Порт:** `5432`
- **Пользователь:** `process.env.DB_USER || process.env.USER`

### Таблицы

| Таблица | Назначение |
|---------|------------|
| `tasks` | Основные задачи (order, status, deadline, responsible) |
| `subtasks` | Подзадачи, привязанные к задачам через `task_id` |
| `workflow_instructions` | Инструкции по взаимодействию (6 ключей: creation, finplan, fund-request, payments, documents, banks) |
| `migration_steps` | 12 шагов переноса карточки из Glide в 1С |

Все изменения (статусы, порядок drag-and-drop, CRUD операций) сохраняются в БД через REST API.

## Запуск проекта

### Предварительные требования

1. **PostgreSQL** должен быть запущен:
   ```bash
   brew services start postgresql
   ```

2. **База данных** `unf_transition_plan` должна существовать.

### Команды запуска

```bash
# Запуск сервера (порт 3001)
node server/index.js

# Или через npm (обрати внимание: script в package.json указывает на src/index.js, что некорректно)
npm start
```

Сервер доступен по адресу: **http://localhost:3001**

### Остановка

```bash
kill $(lsof -i :3001 -t) 2>/dev/null
```

## API эндпоинты

### Данные

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/api/data` | Получить все данные (tasks, subtasks, workflow, migrationSteps) |

### Задачи (Tasks)

| Метод | Путь | Описание |
|-------|------|----------|
| `PUT` | `/api/tasks/status` | Обновить статус задачи (`task_id`, `status`) |
| `PUT` | `/api/tasks/reorder` | Обновить порядок задач (массив `{id, order}`) |
| `POST` | `/api/tasks` | Создать задачу |
| `PUT` | `/api/tasks/:taskId` | Обновить задачу |
| `DELETE` | `/api/tasks/:taskId` | Удалить задачу |

### Подзадачи (Subtasks)

| Метод | Путь | Описание |
|-------|------|----------|
| `PUT` | `/api/subtasks/status` | Обновить статус подзадачи (`subtask_id`, `completed`) |
| `POST` | `/api/subtasks` | Создать подзадачу |
| `PUT` | `/api/subtasks/:subtaskId` | Обновить подзадачу |
| `DELETE` | `/api/subtasks/:subtaskId` | Удалить подзадачу |

### Инструкции и миграция

| Метод | Путь | Описание |
|-------|------|----------|
| `PUT` | `/api/workflow/:key` | Обновить инструкцию |
| `PUT` | `/api/migration/:stepId` | Обновить шаг миграции |
| `PUT` | `/api/migration/status` | Обновить статус шага миграции |

## Функциональность фронтенда

- **Drag-and-Drop** — перетаскивание задач за иконку ⋮, порядок сохраняется в БД
- **Чекбоксы** — отметка выполнения задач и подзадач
- **CRUD задач** — добавление, редактирование, удаление через модальные окна
- **Инструкции** — клик по карточке инструкции → оверлей с контентом, редактирование по паролю
- **Шаги миграции** — клик по номеру шага → редактирование (по паролю)
- **Статистика** — автоматический подсчёт выполненных/активных задач и процента прогресса
- **localStorage fallback** — если API недоступен, данные кэшируются в localStorage

## Особенности разработки

### Статусы задач

- `pending` — в ожидании (по умолчанию)
- `in-progress` — в работе
- `completed` — выполнено

### Fallback-механизм

При загрузке фронтенд сначала пытается получить данные с сервера. Если API недоступен — используется localStorage. При сохранении данные всегда записываются в localStorage как кэш, а также отправляются на сервер.

### Известная проблема

В `package.json` скрипт `start` указывает на `src/index.js`, но сервер находится в `server/index.js`. Для запуска используй напрямую `node server/index.js`.
